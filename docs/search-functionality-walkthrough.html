Rulecraft code walkthrough

How Search Retrieves Rules and Grounds the Oracle

A visual map of the primitives, database, connection path, and design ideas behind this repo's search functionality.

Executive Shape

Rulecraft uses SQLite lexical retrieval as its reliable backbone. User-facing search starts with SQLite FTS5, falls back to a case-insensitive fuzzy search when needed, and renders matching rule cards. The Oracle now uses hybrid retrieval: FTS5 first, optional Qdrant semantic matches second, then injects the merged rules into Claude's system prompt.

Fallback

A simpler substring search over lowercased title, content, and category. It gives title-prefix matches first, then title contains, then category.

Semantic recall

Qdrant vector search

The Oracle can embed questions with OpenAI text-embedding-3-small, search Qdrant, hydrate rule IDs from SQLite, and merge score-filtered semantic hits after FTS results.

Query Normalization Demo

This mirrors the Rust sanitizer: keep only alphanumeric characters and whitespace, split words, then join them with OR. It makes broad recall the default: any matching term can retrieve a rule.

Sanitized FTS5 query 
fn sanitize_fts_query(query: &str) -> String {
    query
        .chars()
        .filter(|c| c.is_alphanumeric() || c.is_whitespace())
        .collect::<String>()
        .split_whitespace()
        .collect::<Vec<_>>()
        .join(" OR ")
}

Baseline Database

What the app connects to

The default configuration uses DATABASE_URL=sqlite:./rulecraft.db. In this workspace, that file exists at the repo root and is about 217 KB.

The live checked-in database contains 159 rows in rules and 159 rows visible through rules_fts. A local Qdrant index can hold one vector per rule.

Where the corpus comes from

The richer corpus lives in data/rules/*.yaml. The importer parses those YAML files, runs schema setup, then upserts each rule into SQLite.

If the app starts with an empty database and no YAML files are present, runtime setup seeds a small fallback set of 10 rules.

Schema Mechanics

Main table

rules is the authoritative table. It stores the displayable data: title, category, content, source, page, and timestamps.

CREATE TABLE rules (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  category TEXT NOT NULL,
  subcategory TEXT,
  content TEXT NOT NULL,
  source TEXT NOT NULL,
  page INTEGER,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

Search index

rules_fts is an external-content FTS5 table. It indexes selected text columns while pointing back to rules by SQLite rowid.

CREATE VIRTUAL TABLE rules_fts USING fts5(
  title,
  content,
  category,
  content='rules',
  content_rowid='rowid'
);
Why triggers matter: insert, update, and delete triggers keep the FTS table synchronized whenever the main rules table changes. This lets admin edits and YAML imports become searchable without a separate indexing job.

Two Search Surfaces

Rules search page

GET /search accepts up to 500 characters, runs FTS5, falls back to fuzzy search if there are no FTS results, and renders templates/rules/search.html.

Autocomplete is separate: static/js/search.js calls /api/search, which always uses fuzzy search with a limit of 8 suggestions.

Oracle RAG retrieval

POST /scenario/ask accepts up to 2,000 characters, runs FTS5, optionally adds score-filtered vector hits, dedupes, caps context, and passes the returned rules to Claude.

src/search/hybrid.rs owns retrieval merging. src/ai/claude.rs formats each rule as title, content, source, and page inside the system prompt.

Underlying Ideas

Accuracy through locality

The app retrieves from a local official-rules corpus before asking an LLM to reason. That narrows the model's context to known 2024 rule text instead of asking it to remember everything from training.

Low operational weight

SQLite FTS5 gives fast full-text search without requiring Elasticsearch or Postgres. Qdrant remains optional so the app still works without vector infrastructure.

Broad recall first

Joining terms with OR means a query like shield cover spell attack can retrieve rules matching any term. This favors finding something relevant over requiring every word.

Semantic search scoped

Semantic retrieval is active for the Oracle only. Normal /search remains keyword/fuzzy for now, which keeps exact rule lookup predictable.
